# Contributing to dbkit

Thanks for your interest! dbkit is a thin resilience/ergonomics layer over SQLAlchemy Core.

## Ground rules

1. **Never re-implement what SQLAlchemy already does** — pooling, dialects, driver handling.
   We wrap and instrument it, we don't replace it.
2. **No ORM.** No declarative models, sessions, or unit-of-work.
3. **Edit `_async/`, never `_sync/`.** The sync package is generated by `tools/run_unasync.py`.
   Run `make unasync` after touching `_async/`. CI fails on drift.
4. **Pure logic goes in `_core/`.** Error classification, config, routing, mapping, and
   policy decisions must be I/O-free and shared by both frontends.
5. **PostgreSQL-specific features go in `postgres/`** and raise `UnsupportedOperationError`
   on other dialects.

## Workflow

```bash
uv sync --extra dev
make unasync                 # regenerate sync from async
make lint type               # ruff + mypy strict
make test                    # unit suite (no DB)
docker compose up -d db
make integration             # real PostgreSQL, sync + async
```

## Commit / PR

- Keep public API changes documented in `CHANGELOG.md` under `[Unreleased]`.
- Add tests for every behavior change; integration scenarios should run against both
  `Database` and `AsyncDatabase`.
